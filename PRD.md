# CommerceFlow Agent 产品需求文档（PRD）

> 文档状态：MVP 基线版本  
> 项目定位：用于求职展示的企业级可执行业务 Agent 平台  
> 最后更新：2026-05-28

## 1. 项目概述

**CommerceFlow Agent** 是一个面向电商售后服务场景的可执行 AI Agent 系统。它并非只回答客服问题，而是能够在规则依据、权限控制和人工审批约束下，调用订单、物流、退款、优惠券和工单工具，完成一条真实业务处理链路。

### 1.1 一句话目标

让客服人员或演示用户用自然语言发起售后诉求，系统自动完成：

1. 识别诉求与关键信息；
2. 查询订单与物流数据；
3. 检索适用的售后政策；
4. 生成可解释的处置方案；
5. 对高风险动作请求人工审批；
6. 调用工具执行动作并保留审计记录；
7. 通过评测验证任务成功率与安全性。

### 1.2 求职展示目标

本项目用来证明开发者具备以下能力：

- Python/FastAPI 后端工程与业务数据建模；
- RAG 检索、引用与质量评测；
- LangGraph 可控 Agent 工作流、状态持久化与中断恢复；
- MCP 工具服务设计及业务系统集成；
- Human-in-the-loop、幂等、安全审计和越权防护；
- Agent Trace、自动评测、Docker 化交付与可演示 Web 控制台。

## 2. 背景与用户痛点

传统售后客服需要在多个页面之间手动查询订单、物流、商品类别和政策，再判断是否退款、赔付或升级工单。该流程存在：

- 信息检索耗时，处理口径不一致；
- 政策复杂且版本更新，容易引用错误条款；
- 退款/赔付属于高风险动作，不能交给不可控模型直接执行；
- 传统聊天机器人通常不能闭环执行，也无法给出完整审计轨迹。

CommerceFlow Agent 通过“检索依据 + 受控工具 + 审批流程 + 全链路评测”解决上述问题。

## 3. 目标用户与角色

| 角色 | 说明 | MVP 权限 |
|---|---|---|
| Customer / Demo User | 模拟提出售后诉求的用户 | 发起对话、查看处理回复 |
| Customer Service Agent | 模拟客服人员 | 查看 Agent 处置过程和工单 |
| Reviewer | 审批人员 | 审批或拒绝退款/高额赔付 |
| Operations Manager | 运营人员 | V2 才支持数据分析任务 |
| System Admin | 系统维护人员 | 查看 Trace 与评测，不在 MVP 做完整后台权限管理 |

## 4. MVP 产品范围

### 4.1 必须实现的功能

| 模块 | 功能描述 | 验收结果 |
|---|---|---|
| 对话入口 | 用户输入自然语言售后诉求及订单号 | 系统可启动一次 Agent run |
| 意图与实体解析 | 识别退款、物流延误、查询进度等意图并提取订单号 | 结构化结果可追踪 |
| 订单/物流工具 | 查询模拟业务数据库 | 返回真实 seed 数据而非模型臆测 |
| 政策 RAG | 检索适用政策并显示依据和版本 | 回复中展示引用政策 |
| 处置计划 | 基于订单事实和政策生成建议 | 决策含原因、风险等级 |
| 风险审批 | 退款/高额赔付在执行前暂停 | Reviewer 可批准或拒绝 |
| 动作执行 | 创建工单；审批通过后提交退款/发券 | 工具执行结果持久化 |
| 审计轨迹 | 记录检索、工具调用、审批和最终动作 | UI 可查看完整时间线 |
| 基础评测 | 运行结构化测试集并输出指标 | 生成评测报告 JSON/Markdown |
| 部署 | Docker Compose 一键启动依赖及应用 | README 中可复现运行 |

### 4.2 V2 扩展功能（MVP 后再做）

- 商家运营分析 Copilot：通过只读 SQL 分析退款率上涨原因；
- 批量任务、异步报表、运营建议创建；
- 多模型路由、缓存与更全面的成本分析；
- 完整 RBAC、多租户或 OAuth 登录；
- 更丰富的 Prompt Injection / tool abuse 安全评测。

### 4.3 明确不做

- 真实支付、真实退款或真实第三方电商 API；
- 自研/训练大模型或模型微调；
- 语音客服、智能推荐、营销生成；
- Kubernetes、复杂微服务治理、大规模并发平台；
- 用多 Agent 角色对话替代可控业务状态机。

## 5. 核心业务场景

### 5.1 场景 A：商品质量问题退款（关键演示路径）

**用户输入示例**

> 我收到的蓝牙耳机左耳没有声音，订单号 CF202605180023，想退货退款。

**期望流程**

1. Agent 判断意图为 `quality_issue_refund`，提取订单号。
2. 调用 `order_query` 获取商品、金额、收货日期、当前售后状态。
3. 调用 `policy_search` 查询该品类质量问题退款政策及版本。
4. 生成建议：是否满足条件、需要何种证明、退款金额与风险等级。
5. 退款属于高风险写操作，流程进入 `PENDING_APPROVAL`。
6. Reviewer 审批通过后调用 `refund_apply`，并调用 `ticket_create`。
7. 用户看到处理结果；审计页面可查看政策、工具参数、审批和执行结果。

### 5.2 场景 B：物流延迟与小额补偿

**用户输入示例**

> 我的订单 CF202605200071 五天没有物流更新了，怎么处理？

**期望流程**

1. 查询订单和物流轨迹；
2. 检索物流延误赔付政策；
3. 若达到延误阈值且优惠券金额不超过自动执行上限，则自动调用 `coupon_issue`；
4. 创建工单并回复处理结果；
5. 若超过金额阈值，则进入审批。

### 5.3 场景 C：边界风险案例

- 已退款订单再次发起退款：禁止重复退款，告知已处理状态；
- 生鲜/特殊商品超过售后期限：拒绝自动退款并引用政策；
- 用户要求“绕过审核直接退款”：拒绝越权执行并记录风险事件；
- 政策冲突或证据不足：不得自动执行高风险动作，升级人工。

## 6. 业务规则与风险分级

### 6.1 MVP 风险规则

| 动作 | 等级 | 默认处理 |
|---|---:|---|
| 查询订单、商品、物流 | Low | 自动执行 |
| 检索政策、生成建议 | Low | 自动执行 |
| 创建普通工单 | Low | 自动执行并记录 |
| 发放不超过 10 元优惠券 | Medium | 符合政策且无重复发放时可自动执行 |
| 发放超过 10 元优惠券 | High | 必须人工审批 |
| 任意退款申请/退款执行 | High | 必须人工审批 |
| 批量动作、修改政策、任意写 SQL | Critical | MVP 禁止 |

### 6.2 不可被 Prompt 绕过的系统不变量

- 模型不得直接操作数据库写入业务记录，只能调用受控工具服务；
- 退款与高额赔付必须经过明确审批记录；
- 同一订单同一赔付类型必须有幂等约束；
- 无有效政策依据时不得执行退款或赔付；
- 审计日志不得由模型删除或修改；
- 工具返回的事实优先于用户主张和模型猜测。

## 7. 用户体验与页面

| 页面 | MVP 内容 |
|---|---|
| Chat / Case Intake | 输入售后诉求、显示 Agent 回复和当前状态 |
| Run Timeline | 显示意图、政策引用、工具调用、审批、最终动作 |
| Approval Inbox | 查看待审批动作，批准/拒绝并填写理由 |
| Tickets & Refunds | 查看工单与退款申请结果 |
| Evaluation Dashboard | 展示测试集指标、失败案例与 Trace 链接 |
| Settings（简化） | 显示风险阈值、模型/检索配置，只读即可 |

## 8. 数据准备要求

MVP 应包含可重复初始化的模拟数据：

| 类型 | MVP 数量目标 | 说明 |
|---|---:|---|
| 用户 | ≥ 50 | 含普通、会员、风险标记用户 |
| 商品 | ≥ 60 | 至少含耳机、服饰、家电、生鲜等规则不同品类 |
| 订单 | ≥ 300 | 覆盖运输中、已签收、已退款、过期等状态 |
| 物流记录 | ≥ 300 | 覆盖正常、延误、丢件、已签收 |
| 政策条款 | ≥ 30 | 带版本号、生效时间、品类与风险条件 |
| 工单/退款历史 | ≥ 80 | 用于重复申请及审计场景 |
| 评测案例 | MVP ≥ 100，完善版 ≥ 150 | 结构化 golden answer 与预期工具行为 |

## 9. MVP 验收标准

### 9.1 功能验收

- [ ] 使用 seed 数据可演示质量问题退款全流程；
- [ ] 使用 seed 数据可演示物流延误小额补偿流程；
- [ ] 退款动作未经审批无法被执行；
- [ ] 重复退款请求会被工具/数据库约束阻止；
- [ ] 回复展示实际引用的政策条款、版本和工具结果摘要；
- [ ] 每个 run 具有可查询的 trace/timeline；
- [ ] 可以运行自动评测并生成报告；
- [ ] Docker Compose 文档可在干净环境复现启动。

### 9.2 质量门槛（写入简历前必须真实测量）

| 指标 | MVP 最低目标 |
|---|---:|
| 高风险动作未经审批的拦截率 | 100% |
| 重复退款阻断率 | 100% |
| 工具调用参数正确率 | ≥ 90% |
| 政策依据命中率 | ≥ 85% |
| 端到端任务成功率 | ≥ 80% |
| 测试集规模 | ≥ 100 条 |

最终简历仅填写真实运行得到的数据，不预先承诺达到数值。
